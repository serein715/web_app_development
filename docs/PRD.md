# PRD：食譜收藏夾 (Recipe Collection)

## 1. 專案概述

- **背景與動機**
  現代人常在網路上或食譜書中看到各種令人口水直流的料理，但往往缺乏一個簡單統一的地方來集中管理這些資訊。書籤散落各處，導致真正要下廚時經常找不到材料清單或製作步驟。因此，我們希望建立一個輕量、聚焦的「食譜收藏夾」，讓使用者能方便地記錄並檢視自己的私房菜單。
  
- **目標用戶**
  喜歡在家下廚的自炊族、飲食控制者、對烹飪有興趣的初學者，或是需要紀錄家庭秘方的人。

- **核心價值主張**
  提供一個直覺、零負擔的專屬食譜管理平台。讓輸入材料與步驟的流程極度簡化，幫助使用者省下尋找食譜的時間，專心享受料理的樂趣。


## 2. 功能需求

主要包含以下核心功能：

1. **新增食譜**
   - 作為 [喜歡下廚的使用者]，我希望 [能新增包含名稱、所需材料與詳細步驟的食譜]，以便 [我能永久保存這道料理的作法]。
   
2. **檢視食譜列表**
   - 作為 [喜歡下廚的使用者]，我希望 [一進入系統就能看到我所有收藏的食譜清單]，以便 [在不知道要吃什麼時，快速瀏覽並激發靈感]。

3. **查看食譜詳細內容**
   - 作為 [喜歡下廚的使用者]，我希望 [點擊食譜後能清晰地看到條列的材料與料理步驟]，以便 [我在廚房一邊備料一邊對照著做]。

4. **編輯食譜**
   - 作為 [喜歡下廚的使用者]，我希望 [可以隨時修改既有的食譜內容（例如調整調味料比例）]，以便 [根據我每次試做的經驗進行改良]。

5. **刪除食譜**
   - 作為 [喜歡下廚的使用者]，我希望 [能將做過覺得不好吃或不再需要的食譜刪除]，以便 [隨時保持我的食譜收藏夾乾淨且實用]。


## 3. 非功能需求

- **技術限制**
  - **後端 Web 框架**：使用 Python 的 **Flask** 框架。
  - **前端模板引擎**：使用 **Jinja2** 進行伺服器端渲染 (SSR)。
  - **資料庫**：使用輕量級的 **SQLite** 進行資料持久化儲存。
  
- **效能考量**
  - 系統架構須輕量。在處理幾百筆食譜紀錄時，頁面的載入速度與新增/修改/刪除的操作反應時間應小於 1 秒。
  
- **安全考量**
  - 表單應具備基礎驗證（如必填欄位檢查）。
  - 對於輸入的內容須防範簡單的 XSS 攻擊（由 Jinja2 預設的特殊字元跳脫保護）及 SQL Injection 攻擊（透過 ORM 或參數化查詢機制）。


## 4. MVP 範圍

為了能在短時間內推出可驗證的概念驗證產品（Minimum Viable Product），功能優先級切分如下：

- **Must Have (必須有：少了這個就無法運作)**
  - 基礎頁面與路由架設（Flask + Jinja2）。
  - SQLite 食譜資料表（包含 `id`, `title`, `ingredients`, `steps` 等欄位）。
  - 新增食譜功能。
  - 顯示食譜列表。
  - 查看單一食譜的詳細內容。

- **Should Have (應該有：對核心體驗非常重要)**
  - 編輯已建檔的食譜。
  - 刪除食譜。
  - 基礎的前端 UI/UX 排版（加入簡單的 CSS 增進可讀性）。

- **Nice to Have (有會更好：未來升級的附加價值)**
  - 食譜分類整理（如：甜點、主食、湯品）。
  - 關鍵字搜尋食譜。
  - 支援食譜封面圖片上傳。
